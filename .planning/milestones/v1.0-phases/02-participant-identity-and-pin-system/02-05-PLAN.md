  <what-built>
Complete Phase 2 identity system:
- Name entry bottom sheet (vaul drawer) with existing names list and conflict detection
- PIN setup sheet (OTP-style, 4 boxes, digits visible) that fires after name claim
- PIN verify sheet (return visit) with shake animation on failure and "Forgot PIN?" after first failure
- Magic link request sheet (email entry) and email delivery via Resend
- Magic link consume flow: single-use token, 30-min expiry, distinct error pages
- DB-backed session cookie (httpOnly, SameSite=Lax, 7-day expiry)
- Personalized "Welcome back, [Name] — Edit your availability" CTA on event page for active sessions
- Magic link error page at /e/[id]/magic with expired/used/invalid states
  </what-built>
  <how-to-verify>
Start the dev server: `npm run dev` (or use the deployed preview URL if available).

**Test 1 — New visitor name + PIN claim (IDEN-01, IDEN-02, IDEN-09)**
1. Open an event URL in an incognito/private window (no cookies)
2. Confirm: Name sheet slides up from bottom with heading "What's your name?"
3. Enter a name (e.g. "Alice") and tap Continue
4. Confirm: Name sheet closes, a toast appears ("Name claimed — now set your PIN"), PIN sheet opens
5. PIN sheet heading reads "Set a 4-digit PIN" with sub-copy "You'll use this to edit later"
6. Enter 4 digits and tap Set PIN
7. Confirm: Page refreshes and shows single CTA "Welcome back, Alice — Edit your availability"
8. Open DevTools -> Application -> Cookies: confirm `timely_session` is `HttpOnly`, `SameSite=Lax`

**Test 2 — Name conflict detection (IDEN-03)**
1. Open the same event in a second incognito window
2. Enter "Alice" in the name sheet
3. Confirm: Inline error message appears referencing the conflict with a suggestion (e.g. "Alice is taken — try Alice2 or a nickname")
4. Change name to "Alice2" and confirm it is accepted
5. Also test "alice" (lowercase) — should be rejected with same conflict message (case-insensitive check)

**Test 3 — Return visit PIN verify (IDEN-04)**
1. In the original Alice window, delete only the `timely_session` cookie
2. Reload the page
3. Confirm: Name sheet opens (session expired/gone — same as new visitor)
4. Enter "Alice" — confirm the PIN verify sheet opens (not PIN setup — Alice already exists)
5. Enter the correct PIN and confirm session is restored and single personalized CTA shows

**Test 4 — Wrong PIN + rate limiting (IDEN-06)**
1. Delete the session cookie again and reload
2. Enter "Alice" in name sheet
3. In PIN verify sheet, enter "0000" (wrong) 5 times
4. After first wrong attempt: "Incorrect PIN" message appears, "Forgot PIN?" link becomes visible
5. After 5 wrong attempts: 429 lockout message appears mentioning 15 minutes

**Test 5 — Magic link flow (IDEN-07, IDEN-08)**
Prerequisites: `RESEND_API_KEY` is set in `.env.local`, and a real email address is available.
1. After seeing "Forgot PIN?", click it and confirm the magic link sheet opens
2. Enter a real email address and tap Send magic link
3. Confirm: Sheet closes, toast confirms "Check your email — a link is on its way"
4. Check the email inbox — confirm an email arrives with an "Access my availability" button
5. Click the magic link in the email
6. Confirm: Redirected to the event page with single CTA "Welcome back, Alice — Edit your availability" — no PIN reset required (IDEN-08)
7. Click the magic link again in the email
8. Confirm: Redirected to /e/[id]/magic?error=used with "This link has already been used" message and a "Request a new link" CTA

**Test 6 — Expired magic link (IDEN-08)**
Skip if the 30-minute wait is impractical. Alternatively:
1. Directly visit `/e/[id]/magic?error=expired`
2. Confirm: "This link has expired" heading and "Request a new link" CTA

**Test 7 — Database verification (SECR-01, SECR-02)**
Run these Drizzle queries (or use `npm run db:studio`):
1. Check participants: `SELECT name, pin_hash, email FROM participants WHERE event_id = '[id]'`
   - `pin_hash` must start with `$argon2` (Argon2id hash)
   - `email` must be NULL after magic link consumption (or never set if magic link wasn't used)
2. Check magic_tokens: `SELECT token_hash, used_at, expires_at FROM magic_tokens`
   - `token_hash` must be a 64-char hex SHA-256 hash
   - `used_at` must be set after magic link consumption
